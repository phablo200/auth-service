# Auth Service Terraform Lab

This Terraform project provisions a lab AWS deployment for `auth-service`.

Architecture:

```text
Internet
  -> Application Load Balancer, HTTP only
  -> ECS Fargate task running auth-service
  -> RDS PostgreSQL
  -> Secrets Manager
  -> CloudWatch Logs
```

It intentionally does not create Route 53, ACM, HTTPS, SSM Parameter Store, Redis/ElastiCache, GitHub Actions, NAT Gateway, or VPC endpoints.

## Requirements

- Terraform `>= 1.6`
- AWS CLI
- Docker
- AWS credentials with permission to create VPC, ALB, ECS, ECR, RDS, IAM, Secrets Manager, CloudWatch Logs, and AWS Budgets resources

## Configure Variables

Create a local tfvars file from the example:

```bash
cp infra/terraform/environments/prod/terraform.tfvars.example infra/terraform/environments/prod/terraform.tfvars
```

Update:

- `container_image`
- OAuth public/callback values if OAuth will be tested
- mail values if email delivery will be tested

Do not commit `terraform.tfvars`.

## Initialize And Apply

```bash
terraform -chdir=infra/terraform/environments/prod init
terraform -chdir=infra/terraform/environments/prod plan
terraform -chdir=infra/terraform/environments/prod apply
```

The first apply creates the ECR repository and base infrastructure. The ECS task will not run successfully until a valid application image exists in ECR and required Secrets Manager values are populated.

## Build And Push The Image

Use the manual tag convention:

```text
git-<short-commit-sha>
```

Example:

```bash
AWS_REGION=us-east-1
ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
TAG=git-$(git rev-parse --short HEAD)
REPOSITORY_URL="$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com/auth-service-prod"

aws ecr get-login-password --region "$AWS_REGION" \
  | docker login --username AWS --password-stdin "$ACCOUNT_ID.dkr.ecr.$AWS_REGION.amazonaws.com"

docker build -t "$REPOSITORY_URL:$TAG" .
docker push "$REPOSITORY_URL:$TAG"
```

Set `container_image` in `terraform.tfvars` to the pushed image URI and apply again.

## Populate Secrets

Terraform creates placeholder Secrets Manager secrets for application credentials. Populate these before expecting the ECS task to become healthy:

- `auth-service-prod/api-key`
- `auth-service-prod/jwt-secret`
- `auth-service-prod/google-oauth-client-id`
- `auth-service-prod/google-oauth-client-secret`
- `auth-service-prod/github-oauth-client-id`
- `auth-service-prod/github-oauth-client-secret`
- `auth-service-prod/mail-user`
- `auth-service-prod/mail-password`

The database password secret is generated by Terraform:

- `auth-service-prod/db-password`

## Validate

After the ECS service reaches steady state:

```bash
terraform -chdir=infra/terraform/environments/prod output alb_url
curl http://<alb-dns-name>/health
```

Expected response:

```json
{"status":"ok"}
```

Run migrations manually from an approved environment that can reach RDS:

```bash
npm run migrate
```

## Destroy

This lab uses RDS deletion protection disabled and ECR force delete enabled:

```bash
terraform -chdir=infra/terraform/environments/prod destroy
```
